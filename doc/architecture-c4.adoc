== Cel dokumentu

Ten dokument opisuje architekturę systemu `Hexabid` w modelu C4.
Zakres obejmuje:

- kontekst systemowy
- kontenery aplikacyjne i integracyjne
- główne komponenty wewnątrz rdzenia i adapterów

Dokument jest zapisany w AsciiDoc, a diagramy są osadzone jako PlantUML z biblioteką `C4-PlantUML`.

== Założenia

- system jest budowany jako wielomodułowy projekt Maven
- domena aukcyjna pozostaje niezależna od Springa i JPA
- wspólny archetyp `Party` znajduje się w rdzeniu i reprezentuje uczestnika biznesowego niezależnie od mechanizmu logowania
- uwierzytelnianie jest adapterem wejściowym, a tożsamość użytkownika jest dostarczana do aplikacji przez port
- kontrakty HTTP są definiowane przed implementacją
- adaptery wejściowe i wyjściowe komunikują się z `auctions-core` wyłącznie przez porty

Na poziomie *System Context* system może korzystać z zewnętrznych dostawców tożsamości (Google, GitHub) lub wewnętrznego mechanizmu dla celów deweloperskich (Local Auth). Wybór konkretnej metody zależy od wariantu budowy aplikacji.


Na poziomie *System Context* nie pokazuję `Job` jako osobnego inicjatora, ponieważ w tej aplikacji scheduler jest elementem wewnętrznym systemu, a nie zewnętrznym aktorem lub osobnym systemem. Dlatego pojawia się dopiero na poziomie *Container*.

== Główne scenariusze

=== Tworzenie aukcji

1. Sprzedający wywołuje REST `POST /api/auctions`.
2. `auctions-adapter-in-rest` pobiera aktualną tożsamość z `CurrentUserProvider` i mapuje request na `CreateAuctionCommand`.
3. `CreateAuctionService` sprawdza KYC przez port `KycClient`.
4. `CreateAuctionService` zwraca jawny wynik utworzenia albo biznesowy powód odrzucenia.
5. `AuctionRepository` zapisuje nową aukcję.
6. REST zwraca widok aukcji zgodny z kontraktem OpenAPI.

=== Licytacja w czasie rzeczywistym

1. Kupujący wysyła ofertę przez WebSocket.
2. Adapter WebSocket pobiera aktualną tożsamość użytkownika z portu auth.
3. `PlaceBidService` wywołuje agregat i zwraca `BidAccepted` albo `BidRejected`.
4. `Auction` waliduje reguły licytacji i aktualizuje historię ofert.
5. `AuctionRepository` zapisuje nowy stan agregatu z optimistic locking.
6. `AuctionEventPublisher` publikuje zdarzenie zmiany lidera do Kafki i do kanału WebSocket.
7. W przypadku konfliktu współbieżności use case zwraca wynik biznesowy `CONCURRENT_MODIFICATION`, a system nie traci stabilności.

=== Zamknięcie wygasłych aukcji

1. Scheduler uruchamia `CloseExpiredAuctionsUseCase`.
2. Core pobiera wygasłe aukcje z `AuctionRepository`.
3. `Auction` zmienia status na `CLOSED` i rozstrzyga, czy aukcja została wygrana czy zamknięta bez zwycięzcy.
4. W razie konfliktu optimistic lock aukcja jest pomijana, a use case zlicza konflikt zamiast zatrzymać cały job.
5. Adaptery publikują biznesowe zdarzenie końca aukcji do Kafki i WebSocket.

== Decyzje architektoniczne

- `auctions-core` nie zna Springa, JPA, Kafki ani HTTP.
- `auctions-auth-core` izoluje model użytkownika i port dostępu do aktualnej tożsamości od Spring Security.
- `auctions-bootstrap` pełni rolę composition root i tworzy use case'y jako beany.
- komunikacja REST i KYC jest contract-first, dzięki generatorom OpenAPI.
- uwierzytelnianie odbywa się przez OAuth2/OpenID Connect z dostawcami Google i GitHub.
- licytacja i powiadomienia aukcyjne są prowadzone przez WebSocket, a REST służy do zwykłych operacji i odczytów.
- zdarzenia domenowe opisują konkretny fakt biznesowy, a nie ogólne "coś się zakończyło".
- use case'y zwracają jawne wyniki biznesowe zamiast przeciekać techniczne wyjątki do adapterów.
- zapis aukcji wykorzystuje optimistic locking; konflikty są obsługiwane stabilnie i jawnie.
- pakiety domenowe są zorientowane na język biznesowy, a nie na technologię.
- uwierzytelnianie jest realizowane w modelu "wtyczkowym" (plugin architecture):
** Wybór dostawcy (OAuth2 vs Local) odbywa się w czasie asemblacji za pomocą profilu Maven.
** Kod niepożądanych adapterów (np. deweloperskiego) nie trafia do paczki produkcyjnej.
** Rdzeń aplikacji (`auctions-auth-core`) pozostaje nieświadomy użytej implementacji dzięki portom.

== Mechanizm wtyczek uwierzytelniania

System wspiera dynamiczny wybór mechanizmu uwierzytelniania na etapie budowania aplikacji. Pozwala to na całkowite odseparowanie kodu deweloperskiego od produkcyjnego.

=== Składanie w czasie budowy (Maven Profiles)

W module `auctions-bootstrap/pom.xml` zdefiniowano profile Maven, które warunkowo dołączają odpowiednie adaptery jako zależności:

* `auth-oauth` (aktywny domyślnie): Dołącza `auctions-adapter-in-auth-oauth`.
* `auth-local`: Dołącza `auctions-adapter-in-auth-local`.

Dzięki temu, budując aplikację poleceniem `mvn clean package -P auth-local`, otrzymujemy artefakt JAR, który fizycznie wewnątrz `BOOT-INF/lib` nie posiada bibliotek związanych z OAuth2, a zawiera kod logowania lokalnego.

=== Automatyczna rejestracja (Spring Component Scanning)

Aplikacja wykorzystuje mechanizm `scanBasePackages = "com.acme.auctions"`. Ponieważ oba adaptery (`oauth` i `local`) znajdują się w podpakietach tej ścieżki, Spring Boot automatycznie wykrywa i rejestruje:

1. **Konfigurację Security**: Klasy oznaczone `@Configuration` (np. `LocalSecurityConfiguration` lub `OAuth2SecurityConfiguration`). 
2. **Implementację Portów**: Klasy implementujące `CurrentUserProvider`.

Jeżeli na ścieżce klas (classpath) znajdzie się tylko jeden z tych modułów, Spring utworzy beany tylko dla niego. W przypadku znalezienia obu (co jest możliwe w środowisku IDE bez poprawnie ustawionych profili), mechanizm Springowy może wymagać użycia `@ConditionalOnMissingBean` lub `@Primary`, jednak w docelowym artefakcie (buildzie) znajduje się zawsze tylko jedna wtyczka.

=== Ekran logowania

Spring Security automatycznie generuje stronę `/login` na podstawie aktywnych filtrów:
* Gdy aktywny jest `formLogin()`, generowany jest formularz logowania.
* Gdy aktywny jest `oauth2Login()`, generowana jest lista dostawców (GitHub, Google).
