== Project Vision

Hexabid is a reference auction platform implementing **hexagonal architecture** (ports and adapters) with **contract-first** development approach.
The system demonstrates enterprise-grade patterns for real-time bidding, payment processing, and clean domain separation.

== Architecture Overview

=== Hexagonal Architecture

The system follows the hexagonal (ports and adapters) architecture pattern:

* **Core Domain** (`auctions-core`) - Pure Java business logic, no framework dependencies
* **Input Adapters** - REST, WebSocket, Job scheduler (inbound)
* **Output Adapters** - JPA, Kafka, KYC, Payment gateways (outbound)
* **Ports** - Interfaces defining boundaries between core and adapters

== Core Domain Models

=== Party (User)

* `Party` - Business participant (buyer/seller) independent of authentication mechanism
* `PartyId` - Unique identifier derived from identity provider
* `PartyRole` - Seller, Buyer, Admin

=== Auction

* `Auction` - Aggregate root with bidding rules
* `AuctionId` - Unique auction identifier
* `AuctionStatus` - OPEN, CLOSED
* `Bid` - Individual bid with amount, currency, timestamp

=== Lot (Item for Sale)

* `Lot` - Represents item being auctioned
* `LotId` - Unique lot identifier
* `SellingMode` - WHOLE, DIVISIBLE, DIVISIBLE_ONLY

=== Product

* `Product` - Product definition in catalog
* `ProductId` - Unique product identifier

== Business Requirements

=== User Registration and Profile

* Users authenticate via OAuth2/OIDC (Google, GitHub) or local credentials
* Profile includes identity verification status (KYC)

=== Auction Creation

* Seller creates auction with title, description, starting price, end time
* KYC verification required before creating auctions
* Real-time validation of business rules

=== Real-Time Bidding

* WebSocket/STOMP for live bidding
* Optimistic locking for concurrent bid handling
* Leader change notifications
* Business result instead of exceptions

=== Payment Processing

* Multiple payment gateways (PayU, Przelewy24, Crypto)
* Currency conversion support
* Accounting entries per transaction

=== Event Publishing

* Domain events to Kafka
* WebSocket notifications to clients
* Event types: AuctionWon, AuctionClosedWithoutWinner, AuctionLeaderChanged

== API Specifications

=== REST Endpoints

* `POST /api/auctions` - Create auction
* `GET /api/auctions/{id}` - Get auction details
* `GET /api/auctions` - Browse/search auctions

=== WebSocket Channels

* `/app/auctions/{id}/bids` - Place bid
* `/topic/auctions/{id}/bids` - Accepted bids
* `/topic/auctions/{id}/events` - Domain events
* `/topic/auctions/{id}/errors` - Business rejections

=== Authentication

* OAuth2/OpenID Connect with Google and GitHub
* Local form login for development

== Non-Functional Requirements

* **Scalability** - Handle concurrent bidding with optimistic locking
* **Security** - OAuth2/OIDC authentication, no credential storage
* **Modularity** - Independent modules with clear boundaries
* **Contract-First** - OpenAPI definitions before implementation

== Development Principles

* Domain logic in pure Java (no Spring/JPA in core)
* Explicit business results instead of exceptions leaking to adapters
* Contract-first with OpenAPI generator
* Event-driven with Kafka and WebSocket
* Plugin architecture for authentication (profile-based)
